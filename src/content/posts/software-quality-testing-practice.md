## 测试计划

### 1. 项目概述

#### 1.1 选题背景

本项目选择 EasyBook 网上书店系统 作为课程作业对象，主要基于以下考虑：

1. 业务流程完整、测试场景丰富：系统覆盖注册登录、图书浏览、购物车、下单、订单管理、后台库存管理等典型电商流程，天然适合功能测试、流程测试与异常测试。
2. 前后端分层清晰，便于分工与协作：后端采用 Spring Boot（Java），前端采用 Vue 3 + TypeScript，适合团队并行开展单元测试、接口测试、组件测试和场景测试。
3. 存在较多分支与边界条件：如登录鉴权、库存校验、订单状态流转、参数合法性判断等，适合使用边界值、等价类、场景法和独立路径法系统设计高质量测试用例。
4. 适合验证 AI 辅助测试的有效性：在被测代码已基本成型的前提下，通过 AI 生成测试代码初稿，再进行人工审查与增强，能够体现“效率提升 + 质量保障”的实践价值。

因此，该题目能够较好支撑课程对“测试方法应用、覆盖率达标、AI 协同开发”的综合要求。

#### 1.2 技术栈选择

本项目测试实践采用如下技术栈：

| 类别             | 技术选型                             | 选择原因                                                                   |
| ---------------- | ------------------------------------ | -------------------------------------------------------------------------- |
| 后端语言         | Java 21（与报告环境一致）            | 与现有 Spring Boot 工程兼容，生态成熟,便于使用标准测试工具链。             |
| 前端语言         | TypeScript + Vue 3                   | 具备类型约束，利于编写可维护的测试与断言。                                 |
| 后端测试框架     | JUnit 5 + Mockito + Spring Boot Test | 支持单元测试、Mock 隔离和接口层测试,便于覆盖 service/controller 多层逻辑。 |
| 前端测试框架     | Vitest + Vue Test Utils              | 与 Vite 项目天然集成,执行速度快,适合组件与 store 测试。                    |
| 覆盖率工具(后端) | JaCoCo                               | 与 Maven 集成方便，可输出行覆盖率、分支覆盖率报告。                        |
| 覆盖率工具(前端) | Vitest Coverage (Istanbul/V8)        | 支持 TS/Vue 覆盖率统计,报告可视化强，便于定位薄弱分支。                    |
| 静态分析         | ESLint(前端)                         | 保障测试代码与业务代码基本规范，减少低级错误。                             |
| 构建工具         | Maven(后端)、npm(前端)               | 与现有项目结构一致,便于 CI/本地统一执行。                                  |
| AI 工具          | GitHub Copilot, GPT 等               | 用于批量生成测试代码初稿、边界数据、异常分支与测试脚手架。                 |

#### 1.3 AI 使用策略

模式选择：模式 B（被测代码人工编写 + 测试代码 AI 生成）。

选择理由：

1. 本项目业务代码已具备一定规模，适合聚焦测试质量提升而非功能再开发。
2. 模式 B更贴合课程目标：验证 AI 在测试用例生成中的效率与质量，并通过人工增强保证正确性。
3. 有利于形成可复用流程：AI 批量生成初稿 → 人工审查修正 → 覆盖率反馈补测。

AI 使用范围：

1. 按模块生成单元测试初稿（后端 service、controller；前端 store、view、router）。
2. 自动生成边界值与等价类输入数据组合。
3. 基于控制流提示生成独立路径测试草案。
4. 为核心业务链路生成场景法测试步骤（如“登录-加购-下单-查单”）。

##### 人工介入点：

1. 测试设计把关：人工确定关键业务、风险点和优先级，不直接照搬 AI 方案。
2. 代码审查修正：检查断言有效性、Mock 合理性、异常分支是否真实可触发。
3. 覆盖率闭环：依据 JaCoCo/Vitest 报告补齐漏测分支，重点修复“只测主流程不测异常流”的问题。
4. 结果复核：随机抽样回归，确保测试通过并与业务需求一致。

##### 禁止行为：

1. 直接提交未经人工审查的 AI 生成测试代码；
2. 为追求覆盖率编写无业务意义的空断言测试；
3. 篡改或规避覆盖率统计口径（如随意排除核心业务类）。

#### 1.4 AI 提示词策略

为保证模式 B落地质量，提示词按阶段分为四类，并统一采用“上下文 + 约束 + 输出格式”的模板。

##### 1. 测试生成提示词（初稿阶段）

目标：快速生成可运行的测试代码骨架。

内容要素：被测类/组件、方法签名、依赖、需要覆盖的分支、期望断言。

输出要求：直接输出测试函数，命名清晰，包含 Given-When-Then 结构。

##### 2. 边界与等价类提示词（数据增强阶段）

目标：补齐输入空间，减少漏测。

内容要素：字段约束（长度、范围、非空、枚举）、合法类与非法类定义。

输出要求：以“测试点表 + 对应断言建议”输出，便于人工筛选。

##### 3. 场景与路径提示词（深度覆盖阶段）

目标：覆盖关键业务链路和复杂分支。

内容要素：业务主流程、失败流程、异常流程、状态变化。

输出要求：给出场景步骤、前置条件、期望结果，以及对应代码路径说明。

##### 4. 审查与修正提示词（质量控制阶段）

目标：发现伪测试、弱断言和不可达分支。

内容要素：现有测试代码、覆盖率报告、失败日志。

输出要求：指出问题清单、修正建议、优先级（高/中/低）。

##### 统一使用规范：

每次提示词必须绑定具体文件与函数，避免泛化输出；

生成后必须由人工核对业务语义与断言有效性；

修改后再次运行测试与覆盖率统计，形成可追踪迭代记录。

### 2. 被测模块说明

#### 2.1 模块划分与职责

结合当前项目代码结构（backend/service、backend/controller、frontend/stores、frontend/views、frontend/router），本组将被测对象划分为5 个测试模块，与 5 位成员一一对应。每位成员负责 1 个主模块 + 0-1 个协同子模块的单元测试与覆盖率提升。

本组选择这些模块作为测试重点，主要原因如下：

1. 分支密度高：认证、订单、库存模块包含大量条件判断与状态流转（如权限、库存、订单状态），对分支覆盖率影响最大。
2. 业务风险高：登录鉴权、下单支付、发货扣库存等链路一旦出错会直接影响核心业务。
3. 前后端联动强：service + store + router/view 可以形成完整场景，便于覆盖场景法与独立路径法。
4. 适配团队并行：5 个模块可对应5 名成员并行推进，降低冲突并提高迭代速度。

具体模块与职责如下：

##### 1. M1 认证与账户模块（负责人：A）

关联范围：backend/service/AuthService.java、frontend/src/stores/auth.ts、frontend/src/router/index.ts。

核心职责：注册、登录、资料维护、改密、注销、路由鉴权与管理员权限控制。

测试重点：账号状态分支（正常/冻结/注销）、密码校验分支、未登录与管理员路由守卫分支。

##### 2. M2 图书检索与详情模块（负责人：B）

关联范围：backend/service/BookService.java、frontend/src/stores/books.ts、frontend/src/views/books/*。

核心职责：图书分页查询、筛选排序、详情查看、图书增删改（管理员）、搜索历史维护。

测试重点：分页参数边界、筛选条件组合、ISBN 不存在分支、新增图书重复校验。

##### 3. M3 购物车与下单模块（负责人：C）

关联范围：backend/service/CartService.java、 backend/service/OrderService.java、frontend/src/stores/cart.ts。

核心职责：购物车增删改查、订单创建、支付、取消、确认收货、金额计算与优惠折扣。

测试重点：空购物车分支、库存不足分支、退款回滚分支、购物车数量边界。

##### 4. M4 库存/缺书/采购模块（负责人：D，协同：E）

关联范围：backend/service/InventoryService.java、backend/service/OutOfStockService.java、backend/service/PurchaseService.java、frontend/src/views/admin/*。

核心职责：入库/出库/盘点、低库存预警、缺书登记与状态流转、采购单处理。

测试重点：in/out/check三类分支、可用库存不足分支、预留库存存在时禁止删除分支。

##### 5. M5 后台订单与客户管理模块（负责人：E，协同：C）

关联范围：backend/controller/*Admin*.java、backend/service/OrderService.java、frontend/src/views/admin/AdminOrdersView.vue、frontend/src/views/admin/AdminCustomersView.vue。

核心职责：管理员查看全量订单、发货、后台删单、客户列表与状态管理、权限隔离。

测试重点：管理端分页与状态筛选分支、发货异常映射分支、仅特定状态可删单分支。

##### 测试范围边界说明（避免“漏测”误解）：

dto/pojo/types/assets/plugins 作为数据结构或配置资源，不单独设为核心模块，但会在业务测试中被间接覆盖；

mapper 主要通过 service 层单元测试中的 mock 隔离验证，必要时以少量集成测试补充；

components/common 与非核心页面以冒烟或基础交互测试为主，不作为覆盖率冲刺主战场。

分工约束说明：每位成员仅负责 1~2 个模块的单元测试；当前方案中每人 1 个主模块，必要时由相邻模块成员协同补测高风险分支。

为保证模块间协作清晰，统一采用以下“主责+协同”口径：

OrderService：C 主责（下单/取消/购物车联动），E 协同（后台订单管理与发货异常映射）；

Inventory/OutOfStock/Purchase：D 主责（库存、缺书、采购主流程），E 协同（采购与后台管理联动场景）；

前端管理页与 api/client.ts：E 主责，D/C 分别在库存与订单联动场景下补测关键分支；

模块报告允许交叉覆盖，但最终汇总按“主责模块达标 + 协同模块补强”验收。

#### 2.2 核心功能列表

下面给出每位成员的核心测试对象（每人 4 个），并以“功能说明 + 输入输出 + 复杂度 + 测试重点”方式描述，作为后续单元测试与覆盖率冲刺的直接任务清单。

##### 1. A（M1 认证与账户模块）

##### 1. AuthService.register

功能说明：完成用户注册并校验用户名是否重复。

主要输入：RegisterRequest（用户名、密码、邮箱、手机号等）。

返回类型：`ApiResponse<RegisterResponse>`。

复杂度：中。

测试重点：用户名已存在分支、正常注册分支（含密码加密与默认字段）。

##### 2. AuthService.login

功能说明：完成登录校验并返回 token 与用户信息。

主要输入：LoginRequest（用户名、密码）。

返回类型：`ApiResponse<LoginResponse>`。

复杂度：高。

测试重点：用户不存在、密码错误、冻结账户、注销账户、登录成功分支。

##### 3. AuthService.changePassword

功能说明：验证旧密码后更新新密码。

主要输入：customerId、ChangePasswordRequest。

返回类型：`ApiResponse<String>`。

复杂度：中。

测试重点：用户不存在、旧密码错误、修改成功分支。

##### 4. router.beforeEach

功能说明：根据登录态与角色执行路由访问控制。

主要输入：to、from、next，以及登录状态/管理员标识。

返回行为：路由跳转决策（重定向或放行）。

复杂度：高。

测试重点：requiresAuth、requiresGuest、requiresAdmin三类分支。

##### 2. B（M2 图书检索与详情模块）

##### 1. BookService.listBooks

功能说明：图书分页查询、筛选、排序。

主要输入：

page/pageSize/keyword/categoryId/publisherId/minPrice/maxPrice/status/sort。

返回类型：`ApiResponse<PageResult<Book>>`。

复杂度：高。

测试重点：分页参数边界、关键词查询分支、价格区间边界、空结果分支。

##### 2. BookService.getBookDetail

功能说明：查询图书详情并聚合库存、作者、出版社信息。

主要输入：isbn。

返回类型：`ApiResponse<BookDetailResponse>`。

复杂度：中。

测试重点：图书不存在分支、库存状态映射（out_of_stock/low/normal）。

##### 3. BookService.createBook

功能说明：新增图书并建立作者、关键词、库存等关联信息。

主要输入：CreateBookRequest。

返回类型：`ApiResponse<Book>`。

复杂度：高。

测试重点：请求为空、ISBN缺失、ISBN重复、作者/关键词创建分支。

##### 4. useBooksStore.add_search_history

功能说明：维护前端搜索历史并持久化。

主要输入：keyword。

返回行为：更新 store 与 localStorage。

复杂度：低。

测试重点：空字符串过滤、去重、仅保留最近 10 条。

##### 3. C（M3 购物车与下单模块）

##### 1. CartService.addItem

功能说明：向购物车新增商品或累加已有条目数量。

主要输入：customerId、isbn、quantity。

返回类型：`ApiResponse<Map<String, Object>>`。

复杂度：中。

测试重点：新条目分支、已有条目累加分支、默认数量分支。

##### 2. CartService.updateItem

功能说明：修改购物车条目数量。

主要输入：customerId、itemId、quantity。

返回类型：`ApiResponse<Map<String, Object>>`。

复杂度：中。

测试重点：条目不存在返回404、数量更新成功分支。

##### 3. OrderService.create

功能说明：从购物车创建订单并处理金额、折扣、库存预留。

主要输入：收货地址、收件人、手机号、支付方式、备注（以及购物车上下文）。

返回类型：`ApiResponse<Map<String, Object>>`。

复杂度：高。

测试重点：购物车为空分支、库存不足转processing分支、正常下单分支。

##### 4. OrderService.cancel

功能说明：取消订单并处理退款与预留库存释放。

主要输入：orderId。

返回类型：`ApiResponse<String>`。

复杂度：高。

测试重点：订单不存在、已发货不可取消、已支付退款与库存回滚分支。

##### 4. D（M4 库存/缺书/采购模块）

1. InventoryService.list

功能说明：分页返回库存列表并计算状态。

主要输入：page/pageSize/keyword/status/sort。

返回类型：`ApiResponse<PageResult<Inventory>>`。

复杂度：中。

测试重点：状态筛选分支、排序分支、分页边界。

##### 2. InventoryService.update

功能说明：执行入库、出库、盘点操作。

主要输入：isbn、operation、quantity、reason、locationCode。

返回类型：`ApiResponse<String>`。

复杂度：高。

测试重点：in/out/check/default分支、可用库存不足分支、回写位置分支。

##### 3. InventoryService.deleteInventory

功能说明：库存软删除并将图书标记为下架。

主要输入：isbn。

返回类型：`ApiResponse<String>`。

复杂度：中。

测试重点：库存不存在、图书不存在、预留库存>0禁止删除、删除成功分支。

##### 4. InventoryService.lowStock

功能说明：返回低库存预警列表并计算状态字段。

主要输入：无。

返回类型：`ApiResponse<List<Inventory>>`。

复杂度：中。

测试重点：库存状态映射、空列表分支。

##### 5. E（M5 后台订单与客户管理模块）

1. OrderService.listAll

功能说明：管理员分页查询全量订单。

主要输入：page、pageSize、status。

返回类型：`ApiResponse<PageResult<Order>>`。

复杂度：中。

测试重点：分页参数兜底、状态筛选分支。

##### 2. OrderService.ship

功能说明：管理员发货并映射存储过程异常。

主要输入：orderId、deliveryCompany、trackingNo。

返回类型：`ApiResponse<Map<String, Object>>`。

复杂度：高。

测试重点：订单不存在、未支付、余额不足、超透支额度、成功发货分支。

3. OrderService.delete

功能说明：管理员删除订单（仅允许已送达）。

主要输入：orderId。

返回类型：`ApiResponse<String>`。

复杂度：中。

测试重点：订单不存在、非delivered拒绝删除、成功删除分支。

4. useAuthStore.fetch_user_profile（管理端鉴权协同）

功能说明：刷新当前用户资料，用于后台权限与展示联动。

主要输入：无（依赖token）。

返回类型：`Promise<User>`。

复杂度：中。

测试重点：成功更新用户、失败抛错、is_loading状态切换。

##### 任务达标总结：

团队共 5 人，每人 4 个核心函数/类，共 20 个核心测试对象；

每人负责 1 个主模块（必要时协同 1 个子模块）；

第 4 章测试用例设计将围绕上述 20 个对象，分别落地边界值、等价类、场景法、独立路径测试。

#### 2.3 测试对象范围与执行口径

测试对象采用“核心清单 + 扩展清单”方式组织，以兼顾基础覆盖与风险补测，统一口径如下：

##### 1. M1（A）补充对象：在

register/login/changePassword/router.beforeEach 基础上，新增getUserProfile/updateProfile/deleteAccount，形成账户全生命周期覆盖。

2. M2（B）补充对象：除 BookService 核心方法外，纳入 useBooksStore 搜索历史、App 渲染与初始化相关测试，强化前端状态与展示层验证。
3. M3（C）补充对象：在 CartService、OrderService 单元测试外，补充 API 级脚本化集成链路（登录→加购→下单→取消）与数据库一致性校验。
4. M4（D）补充对象：覆盖 InventoryService、OutOfStockService、PurchaseService 的组合流程，形成库存-缺书-采购闭环。
5. M5（E）补充对象：聚焦后台订单/客户管理、api/client.ts 错误映射、覆盖率门禁与静态分析治理。

### 3. AI 提示词设计

#### 3.1 提示词设计原则

为保证模式 B可控落地，本组提示词设计遵循以下原则：

1. 目标唯一原则每条提示词只解决一个明确任务（如“生成AuthService.login单元测试”），避免一次要求“生成+优化+重构+总结”。
2. 上下文充分原则必须给出被测文件路径、函数签名、依赖对象、已知业务规则、预期断言，避免AI凭空假设。
3. 覆盖导向原则每条提示词明确要求包含正常流、异常流、边界值和关键分支，优先提升分支覆盖率。
4. 可运行原则输出必须可直接放入当前项目测试框架（后端 JUnit5+Mockito，前端Vitest+Vue Test Utils），禁止伪代码。
5. 可审查原则输出结构统一为“测试点说明 + 测试代码 + 断言理由”，便于人工逐条验收。
6. 最小改动原则AI仅生成或修改测试相关代码，不改业务逻辑；若发现业务缺陷，只输出“问题说明+建议”，由人工决定是否改产品代码。
7. 可追踪原则 每次提示词执行都记录“输入提示词、AI输出、人工修改点、覆盖率变化”，作为作业证据链。

### 4. 测试用例设计

本章基于第2章模块划分与第3章提示词策略，采用AI生成初稿 + 人工修正增强方式设计测试用例。每个模块至少覆盖边界值测试、等价类测试、场景法、独立路径测试四类方法。

#### 4.1 设计目标与实施流程

##### 1. 设计目标

面向 20 个核心函数/类，形成可执行测试用例池；

每个模块至少 1 条完整业务场景链路；

每个核心函数至少覆盖 1 个正常分支 + 1 个异常分支；

四种测试方法均有明确落地样例与代码实现。

##### 2. 实施流程（模式 B）

第一步：由成员根据第3章提示词生成测试初稿；

第二步：人工审查并修正（补强断言、修正mock、补边界）；

第三步：运行测试并查看覆盖率报告；

第四步：按未覆盖分支定向补测，形成第二轮增强；

第五步：沉淀“提示词-生成代码-人工修正-覆盖率变化”证据链。

#### 4.2 等价类测试设计

等价类用于将输入空间划分为“有效类/无效类”，减少冗余用例并保证代表性。

##### 1. M1 认证与账户模块

被测点：AuthService.login

有效类：用户名存在且密码正确；

无效类：用户名不存在、密码错误、账户冻结、账户注销；

关键断言：返回码、错误信息、是否生成token、是否更新最后登录时间。

##### 2. M2 图书模块

被测点：BookService.getBookDetail

有效类：ISBN存在；

无效类：ISBN 不存在/空值；

关键断言：404分支、库存状态映射字段完整性。

##### 3. M3 购物车与订单模块

被测点：OrderService.cancel

有效类：可取消订单（pending/paid）；

无效类：订单不存在、已发货/已送达订单；

关键断言：状态变化、退款处理、预留库存回滚。

##### 4. M4 库存模块

被测点：InventoryService.deleteInventory

有效类：库存存在且无预留库存；

无效类：库存不存在、图书不存在、预留库存>0；

关键断言：删除成功消息、图书状态更新、错误码分支。

##### 5. M5 后台订单模块

被测点：OrderService.delete

有效类：订单状态=delivered；

无效类：订单不存在、状态!=delivered；

关键断言：仅允许已送达订单删除。

#### 4.3 边界值测试设计

边界值用于验证阈值附近行为（最小值、最大值、临界值上下）。

##### 1. 分页边界（M2/M4/M5）

被测点：BookService.listBooks、InventoryService.list、OrderService.listAll

边界输入：page = 0/1、pageSize = 0/1/20/100、null；

预期：触发分页兜底逻辑（默认 page=1,pageSize=20）。

##### 2. 数量与库存边界（M3/M4）

被测点：CartService.addItem、InventoryService.update

边界输入：quantity = 0/1/最大库存/最大库存+1；

预期：合法边界通过，越界触发“可用库存不足”等错误分支。

##### 3. 价格区间边界（M2）

被测点：BookService.listBooks

边界输入：minPrice=maxPrice、minPrice>maxPrice、minPrice=0；

预期：合法区间返回结果；异常区间不崩溃并保持稳定响应。

##### 4. 搜索历史边界（M2前端）

被测点：useBooksStore.add_search_history

边界输入：空字符串、空格、重复关键词、第11 条关键词；

预期：空输入不入库、重复去重、最多保留10条。

#### 4.4 场景法测试设计

场景法强调跨模块业务流程，验证前后端联动与状态一致性。

##### 1. 场景S1：用户下单主流程（A+C）

步骤：注册/登录 → 浏览图书 → 加入购物车 → 创建订单 → 支付订单 → 查看订单详情；

预期：订单状态按预期流转，金额计算正确，前后端数据一致。

##### 2. 场景S2：库存不足处理流程（C+D）

步骤：购物车结算时库存不足 → 订单进入 processing → 触发缺书记录；

预期：订单非失败而转处理中；缺书记录被正确登记。

##### 3. 场景S3：管理员发货流程（E）

步骤：管理员查询订单 → 发货 → 用户确认收货；

预期：状态从 paid -> shipped -> delivered，发货单信息完整。

##### 4. 场景S4：账户安全流程（A）

步骤：登录成功 → 修改密码 → 使用旧密码登录失败 → 新密码登录成功；

预期：密码更新生效，旧凭证失效。

#### 4.5 独立路径测试设计

独立路径法用于覆盖核心函数控制流，确保关键路径至少执行一次。

1. 对象1：OrderService.create（C）

路径P1：购物车为空 → 返回400；

路径P2：有商品且库存充足 → pending下单成功；

路径P3：有商品但部分库存不足 → processing并登记缺书；

路径P4：数据访问异常（mock异常）→ 事务回滚/错误返回。

2. 对象2：InventoryService.update（D）

路径P1：operation=in入库成功；

路径P2：operation=out且库存充足成功；

路径P3：operation=out且库存不足失败；

路径P4：operation=check盘点成功；

路径P5：operation非法进入default分支。

3. 对象3：AuthService.login（A）

路径P1：用户不存在；

路径P2：密码错误；

路径P3：账户冻结；

路径P4：账户注销；

路径P5：登录成功并生成token。

4. 对象4：OrderService.ship（E）

路径P1：订单不存在；

路径P2：订单未支付；

路径P3：余额不足；

路径P4：超过透支额度；

路径P5：发货成功。

### 5. 测试覆盖率目标

本项目覆盖率目标以课程要求为基线，采用“后端 + 前端”双端统计与分阶段达标策略。

#### 5.1 覆盖率指标定义

1. 行覆盖率：统计被测试执行到的有效代码行占比，衡量测试对代码的基本触达程度。
2. 分支覆盖率：统计条件分支（if/else、switch、catch等）被执行的比例，衡量测试对逻辑路径的覆盖程度。
3. 方法覆盖率：统计被调用方法占比，用于评估功能入口是否被测试触达。

本次作业的核心考核指标为“行覆盖率 + 分支覆盖率”，方法覆盖率作为辅助指标用于过程评估。

#### 5.2 覆盖率目标值

##### 1. 总体目标

分支覆盖率（Branch）≥ 85%；

行覆盖率（Line）≥ 90%。

##### 2. 分端目标

后端（backend，JaCoCo）：Line ≥ 90%，Branch ≥ 85%；

前端（frontend，Vitest Coverage）：Line ≥ 90%，Branch ≥ 85%。

##### 3. 模块最低目标

每个主模块（M1~M5）对应核心文件：Line ≥ 85%，Branch ≥ 80%；

若单模块未达标，需在迭代中补测，最终以全项目达标为提交条件。

#### 5.3 统计口径与范围说明

##### 1. 纳入统计范围（核心）

后端：backend/src/main/java/.../service、.../controller、关键 util；

前端：frontend/src/stores、frontend/src/router、关键 views与 api封装。

##### 2. 非核心统计范围（说明性处理）

dto/pojo/types/assets/plugins、纯配置与静态资源不作为覆盖率冲刺主对象；

mapper以service层间接覆盖为主，必要时补充集成测试验证关键SQL路径。

##### 3. 口径一致性要求

不允许为达标而随意排除核心业务类；

所有排除项需在报告中注明理由并经组内评审确认。

#### 5.4 覆盖率采集与检查频率

##### 1. 采集工具

后端：JaCoCo（Maven测试阶段生成覆盖率报告）；

前端：Vitest Coverage（Istanbul/V8报告）。

##### 2. 检查节奏

每位成员每日提交前至少执行1次本模块覆盖率检查；

每两天进行一次全组覆盖率汇总；

里程碑（第一轮、第二轮、冲刺轮）结束时输出对比报告。

##### 3. 输出物要求

覆盖率报告截图或HTML报告存档；

记录“测试新增/调整条数—覆盖率增量—主要补测分支”。

##### 统一要求：

覆盖率结论必须同时给出“原始计数 + 计算公式 + 百分比结果”，例如：

$$
\circ \text {行覆盖率} = \frac {\text {covered}}{\text {covered} + \text {missed}};
$$

分支覆盖率 = $\frac{covered\ branches}{covered\ branches + missed\ branches}$。

前端报告需区分“全项目覆盖率”与“目标文件覆盖率”，避免口径混用。

当存在“单文件100%但全局不足”时，应在报告中单独说明，不得直接替代项目整体结论。

#### 5.5 未达标时的补测策略

若覆盖率低于阈值，按以下顺序补测：

1. 优先补分支覆盖率较低文件（先补 if/else/catch/switch未覆盖分支）；
2. 补关键异常路径（参数非法、库存不足、权限不足、状态不合法）；
3. 补边界值输入（null/空值/0/1/最大值/越界值）；
4. 对场景链路补“失败场景”而非仅成功场景；
5. 通过AI生成补充用例后，必须人工审查并增强断言。

#### 5.6 最终验收标准

项目提交前必须同时满足以下条件：

1. 分支覆盖率与行覆盖率达到课程阈值（Branch ≥85%，Line ≥90%）；
2. 四种测试设计方法（边界值、等价类、场景法、独立路径）均有实际测试代码与执行证据；
3. 完整保留“提示词-AI输出-人工修正-覆盖率变化”的过程材料；
4. 关键业务链路（登录、下单、库存、发货）通过回归测试且结果稳定。

#### 5.7 报告数据质量与口径校验

为避免多成员汇总时出现统计冲突，校验规则如下：

1. 统计一致性：测试方法分布占比总和应为 100%，若存在“同一用例多标签统计”，需在表头明确“可重复计数”。
2. 覆盖率一致性：同一模块在正文、表格、结论中的覆盖率数值必须一致；如不一致，以原始报告文件（CSV/HTML）为准并更正文案。
3. 对象一致性：模块名称与职责须与第2章主责定义一致；出现跨模块对象时必须标注“协同覆盖”。
4. 可追溯性：每个“达标”结论必须能追溯到执行命令、测试文件、覆盖率来源三类证据。

### 4. AI 使用记录

本项目使用到的AI工具包括 OpenAI Codex、Anthropic Claude Code、Microsoft Copilot等。使用 AI 的方式不是简单地让 AI 一次性生成最终测试代码，而是采用“提示词迭代 + 人工校核 + 回归验证”的协作模式。整体流程分为两个阶段。第一阶段为开发与测试执行阶段，AI 主要用于功能分解、风险识别、边界值与等价类用例设计、场景法设计、独立路径设计、自动化测试代码初稿生成和低覆盖分支补测。第二阶段为质量增强阶段，AI 主要用于 SpotBugs、CheckStyle、ESLint 问题分析与修复建议，测试代码重构，flaky 问题排查，以及阶段二验收内容整理。

AI 使用过程中出现过多类典型问题。

第一类是框架生命周期和初始化问题，例如 A 组登录成功分支需要 JWT 密钥初始化，否则纯单元测试环境会抛出异常；修正方式是在测试初始化阶段提供必要配置。

第二类是前端 mock 提升和引用时序问题，例如 Vitest 中 mock 在模块导入前被提升，导致路由守卫或页面组件测试出现初始化顺序错误；修正方式是使用 vi.hoisted、统一 mock 生命周期，并在每个用例后执行mockReset 或类似清理。

第三类是 AI 生成的期望值与真实渲染文本不一致，例如 B 组前端测试初版对 App 展示文本的断言与页面实际内容不一致；修正方式是人工比对源码后校准断言。

第四类是语法或结构错误，例如测试文件中缺失闭合括号；修正方式是人工编译、定位并修补。

第五类是异常文本编码和本地化导致断言不稳定，例如 E 组存储过程异常映射测试如果直接硬编码中文错误消息，可能受编码影响；修正方式是改为从源码提取关键字或断言稳定的错误码。

第六类是覆盖率初始不达标和 flaky 问题，修正方式是根据覆盖率报告补齐低覆盖分支，并固定时间源、控制随机性、隔离外部网络/数据库依赖、统一异步等待策略。

总体评价是，AI 在本次实践中显著提升了测试设计和测试代码初稿生成效率，尤其适合生成边界值、等价类和重复性较强的 mock 测试结构，也能快速定位覆盖率缺口和静态分析问题类型。但 AI 不能替代人工评审。凡是涉及真实业务语义、数据库副作用、前端异步时序、存储过程异常、中文编码、金额与库存一致性的问题，都需要人工结合源码和运行结果进行校准。最终通过率和覆盖率达标并不是 AI 单独完成的结果，而是 AI 生成、人工审查、手工补测、自动化回归和静态分析闭环共同作用的结果。

A 组的典型问题是 JWT 配置初始化依赖运行时上下文。在纯单元测试中，如果登录成功分支没有初始化密钥，测试会抛出异常而不是验证业务逻辑。该问题不属于业务功能缺陷，但会影响测试可执行性，最终通过补充测试初始化逻辑解决。A 组还遇到路由守卫测试中的 Vitest mock 时序问题，初版 mock 引用顺序不稳定，修正为vi.hoisted 后稳定通过。

B 组没有发现图书管理、状态管理和数据库操作的业务功能缺陷，但发现测试代码初稿中存在语法错误、前端期望值与实际渲染内容不匹配、Mock 配置与测试初始化依赖等问题。语法错误通过补齐闭合符号解决；渲染断言通过人工比对 App 文本后修正；Mock 配置问题通过调整时序和引用关系解决。B 组还指出 router 配置、API 客户端等边缘文件覆盖率可进一步加强，这是后续质量提升方向。

C 组的缺陷分析重点在于购物车和订单的一致性风险。购物车侧需要防止新增、累加、修改、删除、清空时出现数量不同步、空值处理异常或汇总金额错误。订单侧需要重点防止空购物车仍创建订单、库存不足导致负库存、部分缺货时状态错误、支付/发货存储过程异常映射错误、取消订单时退款和库存回滚遗漏、并发下单时总售出量超过库存等问题。C 组通过 49 个 JUnit 用例和 REST runner 覆盖了这些风险路径，并给出 DB 校验 SQL，确保响应结果和数据库副作用能够相互印证。

D 组发现的第一个典型业务问题是 InventoryService.update 中出库时可用库存计算未正确考虑预留库存，风险是造成超卖和订单无法履约。修复思路是统一数量变量计算和判空逻辑，并在出库前检查可用库存。第二个问题是 OutOfStockService.register 的缺书记录状态映射不稳定，客户询价和管理员登记可能出现状态不一致，风险是通知状态错误；处理方式是在测试中改为源码关键字提取并稳定断言，同时校准状态映射。第三个问题是 PurchaseService.approve 对非 draft/submitted 状态的采购单审批控制不足，风险是采购单状态流转被破坏；修复方式是增加状态校验。第四个问题是 InventoryService.list 、 OutOfStockService.list和 PurchaseService.list 对 page/pageSize 为 null、负数或 0 时兜底不足，风险是数组越界或返回错误分页数据；修复方式是统一默认分页参数处理。

E 组发现的第一个高风险静态缺陷是 PasswordEncoder.encode 使用 getBytes() 时未指定字符集，属于默认编码依赖问题，可能导致跨环境哈希不一致；修复方式是显式使用 UTF-8。第二个高风险问题是OrderService.create 中存在先解引用再空判断的空值路径，风险是潜在 NPE；修复方式是统一数量变量计算与判空顺序。第三个问题是 OrderService.pay/ship 的异常文本匹配受编码影响，导致分支断言不稳定；修正为测试侧使用源码关键字提取和稳定错误码断言。第四个问题是 AdminOrdersView.spec.ts 与AdminCustomersView.spec.ts 中全局 stub 和 mock 状态可能串扰；修复方式是统一 setup/teardown 、mockReset 和异步等待策略。

剩余风险主要包括三点。第一，前端覆盖率门禁当前更多集中在 src/api/client.ts、路由守卫、状态管理和部分后台页面，库存、缺书、采购等管理页组件虽然已有测试设计，但还可以纳入更严格的覆盖率门禁。第二，采购收货入库、订单支付和发货依赖数据库存储过程或触发器，当前多数测试通过 mock 或本地数据库验证完成，后续可以引入 Testcontainers + MySQL 做更接近生产环境的集成测试。第三，CheckStyle warning 和ESLint warning 尚未完全清零，虽不影响本次作业验收，但属于后续维护中的代码规范技术债。

### 6. 测试结论

从功能正确性看，五个分工模块均完成了核心成功路径、异常路径和关键边界路径的测试。认证与账户模块验证了注册、登录、改密、资料维护、账户注销和路由访问控制；图书检索与详情模块验证了图书查询、详情、CRUD、搜索历史和状态管理；购物车与下单模块验证了购物车操作、订单创建、支付、发货、取消、确认和库存/金额一致性；库存/缺书/采购模块验证了库存状态计算、出入库、低库存预警、缺书生成、采购审批和收货；后台订单与客户管理模块验证了管理端订单全链路和客户管理全链路。所有自动化测试执行结果均为通过，功能正确性结论为当前实现满足本次测试计划中的关键分支要求。

从覆盖率达标情况看，本轮调整后的 A-D 组关键后端类均达到课程设定的 Line ≥ 90%、Branch ≥ 85% 目标。A 组 AuthService 行覆盖率 100.00%、分支覆盖率 96.30%；B 组 BookService 行覆盖率 100.00%、分支覆盖率 86.05%，虽然未能完全复现原报告 95% 的分支覆盖率，但已满足兜底门禁；C 组 OrderService + CartService 行覆盖率 99.22%、分支覆盖率 85.54%；D 组 InventoryService + OutOfStockService + PurchaseService 行覆盖率 99.73%、分支覆盖率 90.91%。E 组后端行覆盖率 99.60%、分支覆盖率 87.06%，前端 API 客户端同样达到 Line 100%、Branch 94.44%。整体来看，本轮 A-D 用例迭代已经满足用户指定的覆盖率底线，并显著提升了认证、图书、购物车/下单、库存/缺书/采购模块的边界分支覆盖。

从代码质量看，阶段二静态分析和测试重构取得了明确效果。SpotBugs 从 11 条高/中优先级问题降至 0，ESLint error 保持 0，warning 从 186 降至 111。测试代码中重复 mock、脆弱断言、异步等待不稳定、时间/随机依赖等问题经过治理后稳定性提升。CheckStyle warning 和部分前端 warning 尚未完全清零，后续仍需持续治理。

从 AI 使用效果看，AI 对本项目的最大价值在于提高测试设计广度和测试代码初稿生成效率。AI 能快速给出边界值、等价类、场景法和独立路径的用例骨架，也能辅助定位覆盖率缺口和静态分析问题。但 AI 对真实业务语义、数据库副作用、前端时序、中文异常编码和金额、库存一致性的理解仍需要人工校准。因此，本次实践形成的有效经验是：AI 适合做“初稿生成、候选分支枚举、问题定位辅助”，人工必须负责“业务正确性判断、断言质量校准、运行验证和最终验收”。

本次测试实践的主要收获包括：团队建立了按模块分工、按测试方法覆盖、按阶段治理质量的测试流程；掌握了JUnit 5、Spring Boot Test、Vitest、Vue Test Utils、JaCoCo、Vitest Coverage、SpotBugs、CheckStyle、ESLint 等工具的组合使用；通过覆盖率报告和静态分析报告驱动补测和修复；通过 AI 提示词迭代提升了测试设计效率。后续改进方向包括将更多前端管理页纳入覆盖率硬门禁，引入 Testcontainers + MySQL 验证存储过程和触发器相关链路，继续降低 ESLint 与 CheckStyle warning 数量，并补充跨模块端到端回归，例如“登录—搜索图书—加入购物车—下单—支付—发货—取消/收货”的完整用户链路。
